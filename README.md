## ターゲットユーザーと提供価値（ビジネスインパクト）

- **ターゲット層:** 毎朝のお弁当作りに追われる、忙しい共働き世代・子育て中の主婦。
- **解決する課題と導入メリット:**
    1. **思考コストの削減:** 「今日何作ろう…」と冷蔵庫の前でフリーズしてしまう、毎日の献立考案ストレスをAI連携で解放します。
    2. **経済的コストの削減とSDGs貢献:** 冷蔵庫内の食材を的確に管理・消費することで、毎月の給食費相当（約400円/日）のコスト削減を実現しつつ、栄養バランスの取れた献立を自動で導き出します。さらに、家庭内の食材廃棄（フードロス）ゼロにも貢献します。

## 開発背景:「チーム開発の課題」と「自身のリアルな悩み」からのフルリメイク

職業訓練校のチーム開発にて「レシピ・食材管理アプリ」を作成した際、**限られた開発期間の中で全体の機能実装を優先した結果**、「データベース設計の甘さからくる在庫連動のエラー」や「UIの統一感の欠如」といった課題が残りました。

また、私自身が2児の母として毎日キッチンに立つ中で、「在庫を管理すること以上に、そもそも毎日ゼロから献立を考えること自体が一番しんどい」と痛感していました。

そこで、「実運用で破綻しない強固なDB設計」を最優先事項としつつ、「忙しいママが一切思考せずに済むAI献立生成」を掛け合わせた完全版として、本プロジェクトを個人開発で立ち上げました。前職（調達・購買部門）で培った10年間の在庫管理ロジックと、当事者目線でのUXを融合させています。

## 技術的な取り組み（開発スコープ）

本アプリは、要件定義・DB設計・API連携・UI実装まで一貫して個人で開発しました。
特に、データ整合性を担保するためのDB設計と、AI出力の不確実性を前提としたAPI設計に注力しています。

## こだわり設計（設計力　　× 技術力アプローチ）

### 1.

前職の購買業務で培った在庫管理・納期管理の知見を家庭の冷蔵庫に応用し、「実運用で破綻しないデータ構造」を重視して設計しました。

- **ステータスによる厳格な分離:** 冷蔵・冷凍の状態をDB上で明確に分離。
- **動的な期限管理（賞味期限アラート）:** Carbonを用いた現在時刻との相対計算により、「今すぐ使い切るべき食材」をダッシュボード上で赤・黄のアラートとして視覚的に警告します。

### 2. AI出力の不確実性を制御する「お弁当特化型AIレコメンド」

生成AI（Gemini API）特有の「出力揺れ」によるシステム不整合を防ぐため、プロンプト設計とレスポンス制御に注力しました。

- **構造化データの強制:** AIの出力を自然言語ではなく厳密なJSON形式に指定し、システム側で安全にパース・保存できるフローを構築。
- **抽出ロジックの工夫:** ユーザーの余り物データから毎回ランダムに4つの食材を抽出しAPIと連携。「子供が食べやすい魔法の工夫」などの条件を加味した最適なレシピを自動提案します。
- **レスポンスのバリデーション:** AIの返答エラーやJSON構文エラーをキャッチする処理を実装し、不正なデータの保存やシステムダウンを防止しています。

### 3. 視覚的ノイズを排除したUIと「栄養・彩りの可視化」

忙しい朝でもユーザーの視覚的なノイズにならないよう、余白を活かした「大人シンプル」で洗練されたアースカラーベースのデザイン設計を徹底しました。

- 各レシピ詳細には「タンパク質やビタミン」等の数値を、Bladeの動的レンダリングを用いて直感的なバーチャートとして可視化。お弁当特有の「見栄えと栄養バランス」の課題を解決します。

### 4. UXを優先した機能削減の意思決定（引き算の設計）

当初は「レシピ作成後、使った食材の在庫を自動で減らす機能」を実装していましたが、AIの出力揺れによる在庫連動エラーのリスクを考慮し、あえて自動減算機能を削除しました。機能を追加することよりも、「ユーザーに誤った情報を提示しないこと（データの信頼性）」を優先した設計判断です。

## 使用技術と選定理由

- **Laravel 10.x (PHP 8.x)**
    - MVC構造により責務分離がしやすく、将来的な機能拡張や保守性を意識した設計が可能なため採用。
- **Blade / Tailwind CSS**
    - コンポーネント指向を取り入れたシンプルな構成でUI構築を高速化し、フロントエンドのUX改善に注力するため。
- **JavaScript (Ajax)**
    - お気に入り登録など、ページ遷移を減らすことでユーザーにストレスのない操作性を実現するため。
- **MySQL**
    - 在庫管理や中間テーブルといった「構造化データの整合性担保」に最も適しているため。
- **Google Gemini API (gemini-2.5-flash)**
    - 高速なレスポンスタイムを活かし、ユーザーを待たせないリアルタイムな献立生成を実現するため。

**【デモ体験の手順】**

1. **アプリURL:** [https://my-kitchen-stock-xxxxx.herokuapp.com](https://my-kitchen-stock-xxxxx.herokuapp.com) にアクセス
2. ログイン画面下部の「デモユーザーでログイン」をクリックしてください。
3. **「ホーム（ダッシュボード）」**にて、期限別のアラート機能をご確認いただけます。
4. **「AI献立提案」**ボタンより、現在の在庫に基づいたレシピ生成をご体験ください。

> 💡 **【運用に関する補足】**
> 自由にデータの追加・更新・削除をお試しください。
> 実際のWebサービス運用を想定し、ゲストユーザーのデータは**毎日深夜0時に自動リセット（Seederの再実行）**される設計としております。
